Hydra Framework

Guía de inicio rápido · Perfil desarrollador · Stack: PostgreSQL · Docker · GitHub Actions

01
Requisitos

Necesitas uno de estos dos entornos. Docker es la opción recomendada.

✓ Docker Desktop 4+ o PostgreSQL ≥ 14 local git bash (Git Bash en Windows) jq (opcional — inject.sh)
En Windows: usa Git Bash o WSL para los scripts .sh. PowerShell no ejecuta bash nativo.
02
Clonar y configurar credenciales

Clona el repo y crea tu archivo de entorno local. Nunca commits el .env.

git clone https://github.com/RubenGZ/Start-up-Pack.git
cd Start-up-Pack

# Crear .env local (gitignored)
cp .env.example .env

Edita .env con tus valores:

DATABASE_URL=postgres://hydra:tu_password@localhost:5432/hydra_dev
POSTGRES_PASSWORD=tu_password_seguro
JWT_SECRET=$(openssl rand -hex 32)

# Solo si usas billing:
STRIPE_SECRET_KEY=<desde dashboard.stripe.com>
STRIPE_WEBHOOK_SECRET=<desde dashboard.stripe.com/webhooks>
03
Levantar la base de datos

Con Docker (recomendado) — PostgreSQL 16 + pg_cron + migrations automáticas:

docker compose up -d

# pgAdmin opcional:
docker compose --profile tools up -d
# → abrir http://localhost:5050

Sin Docker — PostgreSQL local:

createdb hydra_dev
export DATABASE_URL="postgres://usuario:password@localhost:5432/hydra_dev"
bash core/migrations/runner.sh
Output esperado:
=== startup-pack migration runner ===
→ [0001] 0001_init ... OK
→ [0002] 0002_users ... OK
... (24 migrations aplicadas)
04
Inyectar módulos para tu startup

Selecciona solo los módulos que necesitas. El resto no se copia a tu proyecto.

# Inyección completa (todos los módulos)
bash inject.sh --reset --project=MiStartup

# Inyección selectiva (recomendada)
bash inject.sh --reset \
  --modules=auth,users,billing,health-base \
  --project=MiStartup \
  --no-oauth   # excluye Google OAuth si no lo necesitas
base
UUID, pgcrypto, CITEXT, set_updated_at()
auth
Magic link, OAuth Google, sesiones
users
users, organizations, RBAC members
billing
Subscriptions, Stripe webhooks, portal
health-base
health_check() → JSON status
rate-limit
Sliding window por org/endpoint/plan
audit
Trigger universal → audit_log JSONB
onboarding
Progress multi-step + invitaciones
notifications
In-app + email queue + Slack
api-keys
hk_live_* prefix+hash, revoke, usage
dashboard
org_dashboard() — métricas JSONB
seed
Datos demo reproducibles (dev only)
05
Aplicar migrations y rollback

El runner aplica solo las migrations pendientes. Es idempotente.

# Aplicar todas las migrations pendientes
bash core/migrations/runner.sh

# Preview sin ejecutar
bash core/migrations/runner.sh --dry-run

# Revertir hasta la versión N (exclusive)
bash core/migrations/runner.sh --rollback=18
# → revierte 0024 → 0019 (deja 0018 aplicada)
Cada migration tiene su down script en core/migrations/sql/down/. En producción: ejecutar rollback en ventana de mantenimiento con backup previo.
06
Verificar que todo funciona

Conecta al PostgreSQL y ejecuta los smoke tests básicos:

# Conectar a la DB (Docker)
docker compose exec db psql -U hydra -d hydra_dev

-- Smoke test 1: health check
SELECT health_check();
-- → {"status":"ok","version":"0024","timestamp":"..."}

-- Smoke test 2: rate limiting activo
SELECT check_rate_limit(
  '00000000-0000-0000-0000-000000000001'::UUID,
  'api/test'
)->'allowed';
-- → true

-- Smoke test 3: dashboard por org
SELECT org_dashboard(
  '00000000-0000-0000-0000-000000000001'::UUID
)->'billing';
-- → {"plan":"free","status":"active",...}
07
Adaptar a tu startup

Cuando quieras reutilizar Hydra para un nuevo proyecto, usa el flujo ADAPTAR A:

# Reset + inyectar módulos de tu nueva startup
bash inject.sh --reset \
  --modules=auth,users,billing,health-base,audit \
  --project=NuevaStartup

# Aplicar a la DB
bash core/migrations/runner.sh

# active_project/ ahora tiene solo los módulos de NuevaStartup
cat active_project/blueprints.json
Convención de credenciales:
Local → .env (gitignored)  ·  CI → dummy hardcoded  ·  Staging/Prod → variables en Railway / Render / Fly.io. Nunca secrets en el repo.
